# Build Brief — Particle-Collocation Audio ("say the word + particle")

**Status:** design doc / not started. Authored by **Idea CC** (2026-07-05) off Alex's feedback on the shipped particle-cloze card. **Lane: Feature CC** (audio pipeline + card wiring) **+ Alex's paid-run authorization** (ElevenLabs). Scope decided by Alex: **generate proper audio** (not clip-stitching, not show-only).
**Goal in one line:** when a learner answers the particle-cloze card, play a clip that says the **word with its particle** (かさを → "kasa o") — so the particle is heard as part of the collocation, correctly.

> Design doc, not an implementation plan. Grounded in the repo 2026-07-05; repo is source of truth.

---

## Why proper audio (not the cheap path)

The particle card (`ClozeCard` with `particle`, shipped) already plays audio on answer — but it plays the **item's own word clip** (`playIfEnabled()` → `/audio/{lang}/{id}.mp3`), i.e. just かさ ("kasa"), not the particle. Alex wants かさを.

**The cheap fix (stitch word clip + particle's kana clip) is wrong** — it mis-pronounces the two most important particles:
- **は** as a particle is said **"wa"**, but the kana clip は says **"ha"**
- **へ** as a particle is said **"e"**, but the kana clip へ says **"he"**

(を/に/で/と/も are fine.) Stitching would teach the wrong sound for exactly the particles learners struggle with. So we generate a **single collocation clip** instead.

## The insight that makes it correct: one utterance = right pronunciation

Generating **"かさを" / "かさは" as a single ElevenLabs utterance** lets the model read は→"wa" and へ→"e" *automatically*, because it sees the particle in context. This is the same hard-won lesson as the isolated-kana TTS saga (`generate-audio.mjs` header: bare call, context is what makes it accurate). The collocation clip solves は/へ for free — no special-casing.

⚠️ **Verify by ear before trusting it** (same discipline as the kana-TTS rounds): generate a few は/へ collocation clips first and confirm they actually voice as "wa"/"e". If a bare call on `かさは` still says "ha", fall back to giving more context (the fuller example phrase) — but try the bare collocation first, per the pipeline's rule.

## What exists (ground truth — verified 2026-07-05)

- **Clip playback** — `src/store/itemAudio.js` → `useItemAudio(item, {autoplay})` plays `/audio/${item.lang}/${item.id}.mp3`; `playIfEnabled()` (respects the `autoplayAudio` pref) is already called on answer in `ClozeCard`. Missing/failed clip → **silent** (no robot fallback).
- **The card** — `src/components/games/ClozeCard.jsx`, `particle` mode: `blankParticle(item)` + `particleChoices(item, 4)` (`src/store/cardRouting.js`). Routed as `particle:choice` from `Review.jsx` when `shouldParticleCloze(item)`.
- **Generation** — `scripts/generate-audio.mjs`: Haruki voice, `eleven_v3`, **bare call only** (no `language_code`, no kana→katakana, no custom `voice_settings` — those wreck pronunciation). Writes `public/audio/{lang}/{id}.mp3`. Reads `ELEVENLABS_API_KEY` from `.env.local`.
- **Manifest** — `src/data/audioManifest.js` → `AUDIO_IDS` Set (auto-generated by `scripts/generate-audio-manifest.mjs`); the runner routes audio only for ids in it.

## Design (v1)

### 1. Generate collocation clips
- A generation pass (extend `generate-audio.mjs` or a sibling `generate-particle-audio.mjs`) produces, **per particle-cloze-eligible item**, one clip of the collocation `[word][particle]` — **bare call, Haruki, eleven_v3** (identical settings to word clips).
- **Store separately** so it never collides with the word clip: `public/audio/{lang}/particle/{id}.mp3`.
- **Which word?** The token immediately before the blanked particle. **v1 guard (avoids needing the full tokenizer):** only generate when `item.front` sits *directly* before the target particle in `example.jp` → collocation = `front + particle`. Reuse whatever `blankParticle`/`shouldParticleCloze` already use to locate the target particle. Items that don't satisfy the guard simply keep the word-clip behavior (safe degradation).

### 2. Manifest
- Add a `PARTICLE_AUDIO_IDS` Set (extend `generate-audio-manifest.mjs`) so the card knows which items have a collocation clip.

### 3. Wire the card
- In `ClozeCard` particle mode, on answer: if `PARTICLE_AUDIO_IDS.has(item.id)` → play `/audio/{lang}/particle/{id}.mp3`; **else fall back to the current word clip.** Needs a small `useItemAudio` option to play a variant path (e.g. `useItemAudio(item, {autoplay:false, variant:"particle"})` resolving the `particle/` path), or a dedicated play call. Keep the `autoplayAudio`-pref gating (`playIfEnabled`) unchanged.
- **On answer only** (reinforcement) — never on mount (must not reveal the answer).

## ND / correctness constraints
- **Reinforcement, not a gate** — plays after the answer, respects the `autoplayAudio` toggle, silent-safe if a clip is missing.
- **は/へ correctness is the whole point** — the ear-check above is mandatory before shipping; a particle card that says "ha" for a topic-は would actively mis-teach.
- **Bare call only** — do not "fix" pronunciation with `language_code`/`voice_settings`; the collocation context is the fix.

## What this does NOT touch
- ❌ No FSRS / mastery / rung / schema change — `particle:choice` already shipped.
- ❌ No change to the word clips or `AUDIO_IDS`.
- ❌ No new dependency.
- ✅ Only: a generation pass + a `particle/` clip dir + a `PARTICLE_AUDIO_IDS` manifest + a small `ClozeCard`/`useItemAudio` wiring.

## Forcing-function / phases
- **P.0** — pick ~5 items incl. a は and a へ; generate their collocation clips; **ear-check** は→wa, へ→e. Prove the bet before a full run.
- **P.1** — generation pass over all eligible items (guard: front directly precedes the target particle) → `public/audio/{lang}/particle/`; regenerate the manifest (`PARTICLE_AUDIO_IDS`).
- **P.2** — wire `ClozeCard` particle mode to play the collocation clip on answer, fallback to word clip; keep the pref gating.
- **P.3** — Alex feel-check (does it say "kasa o" clearly? は/へ right? fires only after answer?), then merge.

## Open decisions (lock before building)
1. **Clip storage** — `audio/{lang}/particle/{id}.mp3` (recommended) vs an id suffix.
2. **Eligibility guard** — front-directly-precedes-particle only (recommended v1), or invest in the tokenizer to handle "the word before the particle" generally (ties to Batch-2 R11's tokenizer).
3. **Paid run** — one clip per eligible item; **Alex authorizes + runs** `generate` with the API key (as with the 575-clip A1 backfill). Rough count = particle-cloze-eligible items (a subset of the ~448 vocab).
4. **は/へ fallback** — if bare collocation still mis-voices, use the fuller example phrase as context (decide after the P.0 ear-check).

**DoD:** answering a particle-cloze card plays a single clip saying the word with its particle, correctly voicing は/へ, gated by the autoplay-audio pref, with a silent-safe fallback to the word clip when no collocation clip exists. No FSRS/schema change; the particle card kind is unchanged.
