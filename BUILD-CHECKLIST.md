# Lingua ‚Äî Build Checklist & Progress Tracker

**Purpose:** single source of truth for project state. If Alex steps away for a while, opening this cold should answer "what's done, what's next, where did I leave off" in under a minute. Lives at repo root (`BUILD-CHECKLIST.md`); **CC (Claude Code) commits updates** to it as part of every PR.

**Roles live in `CLAUDE.md` ‚Üí "Roles ‚Äî read this first."** Four CC lanes: **Feature CC** (app/engine/UX, draft PRs), **Curriculum CC** (content units, self-merges on green), **QA CC** (tests + reports, builds nothing, never merges), **Idea CC** (ideation ‚Äî feeds the Feature CC backlog, builds nothing, proposes only). Alex merges everything. Three report/route sections in this doc: **Feature CC backlog** (engine/app work surfaced by other lanes), **QA findings** (bugs/defects surfaced by QA, routed to a lane), and **Idea CC ‚Äî feature pitches** (ranked proposals for the Feature/Curriculum lanes to pick up).

## How to use this doc ‚Äî READ FIRST

**If you're Alex, resuming after a break:** read **Status at a glance** below, then jump to the first `[ ]` (or `[!]`) item under the current phase. That's your next action. The checked history above it is what's already shipped. Check **Dead ends & gotchas** before retrying anything that "should work" ‚Äî it may already have failed for a known reason.

**If you're CC, maintaining this doc ‚Äî marking protocol (follow verbatim):**
This file is updated as part of the PR that completes work. When a task is finished (PR merged to `main`), do ALL of:

1. Change `[ ]` to `[x]`.
2. Append ` ‚Äî DONE <YYYY-MM-DD HH:MM local>, PR #<n>`.
   Example: `- [x] Add validateContent ‚Äî DONE 2026-06-21 15:40, PR #11 (CC)`
3. If the work surfaced follow-ups, add them as new `[ ]` items under the correct phase, with a one-line note on *why*.
4. Update the **Status at a glance** block (current phase, last updated, next action).
5. **Never delete** a completed task. The checked list *is* the project memory. Tasks stay in their phase section, checked in place ‚Äî don't move them to a "done pile."
6. Mark work that's actively being built (PR open, not merged) as `[~]` with ` ‚Äî STARTED <date>`. Flip to `[x]` only on merge.
7. Use `[!] ‚Äî BLOCKED ON: <thing>` for anything stuck on a dependency (e.g. Phase 6 is blocked on the Phase 5 backend), so it's never mistaken for a plain todo.
8. When you discover a gotcha that wasn't obvious from the docs ‚Äî a React 19 / Zustand ordering trap, a CI-vs-local difference, a content-contract subtlety ‚Äî append it to **Dead ends & gotchas** so the next session doesn't re-walk it. Bugs fixed *within* a feature PR (like the `useSyncExternalStore` ordering fix in PR #11) should also be noted on the parent item so the history is preserved.
9. **Log everything, even concepts (Alex's standing rule, 2026-06-25).** Anything we add, change, or even just discuss as a future idea goes in this file ‚Äî if it's not actionable now, park it in **Future / come back later** so nothing is lost. Don't rely on memory or chat history; the checklist is the record.
